# 상수, 리터럴, var

## 학습 목표
- `final`의 의미를 변수/필드/참조 관점에서 구분할 수 있다.
- 리터럴 표기 규칙(진법, 접미사, 언더스코어, escape)을 정확히 사용할 수 있다.
- `var` 타입 추론의 장점과 제한 사항을 실무 기준으로 판단할 수 있다.

---

## 1. 상수(Constant)란 무엇인가

상수는 실행 중 값이 바뀌지 않도록 제한한 식별자다.  
Java에서는 보통 `final`로 표현한다.

```java
final int MAX_RETRY = 3;
```

핵심:
- 한 번 대입하면 재대입 불가
- 의도(변하지 않는 값)를 코드에 명시
- 버그 예방과 가독성 향상

---

## 2. `final`의 세 가지 맥락

## 2.1 final 지역 변수

```java
final int timeout = 30;
// timeout = 60; // 컴파일 오류
```

메서드 내부에서 값 안정성을 보장하고 싶을 때 사용한다.

### 2.2 final 필드

```java
class User {
    final String id;

    User(String id) {
        this.id = id; // 생성자에서 1회 초기화
    }
}
```

객체 생성 이후 바뀌면 안 되는 속성에 적합하다.

### 2.3 static final 상수

```java
public static final double VAT_RATE = 0.1;
```

- 클래스 단위로 하나만 존재
- 보통 대문자 + `_` 관례 사용
- 프로젝트 전역에서 공유되는 고정 규칙값에 사용

---

## 3. `final` 참조 변수의 의미

`final`이 참조형에 붙으면 "참조값 재대입 금지"를 의미한다.

```java
final StringBuilder sb = new StringBuilder("A");
sb.append("B"); // 가능(객체 내부 상태 변경)
// sb = new StringBuilder("C"); // 불가(재대입 금지)
```

즉, `final`은 객체의 완전 불변을 자동으로 보장하지 않는다.  
완전 불변을 원하면 클래스 자체를 불변 설계해야 한다.

---

## 4. 컴파일 타임 상수와 런타임 상수

다음처럼 `static final` + 리터럴 조합은 컴파일 타임 상수가 될 수 있다.

```java
public static final int PORT = 8080;
```

반면 런타임 계산 결과는 컴파일 타임 상수가 아니다.

```java
public static final int PORT = Integer.parseInt("8080"); // 런타임 결정
```

컴파일 타임 상수는 `switch case` 라벨, 애노테이션 값 같은 곳에서 중요하다.

---

## 5. 리터럴(Literal) 총정리

리터럴은 코드에 직접 적는 값 자체다.

## 5.1 정수 리터럴

```java
int a = 10;        // 10진
int b = 0xFF;      // 16진
int c = 0b1010;    // 2진
int d = 012;       // 8진(주의)
long e = 3_000_000_000L;
```

주의:
- `L` 대신 소문자 `l`은 숫자 `1`과 헷갈리므로 지양
- 언더스코어 `_`는 자리 구분용이지만 시작/끝, 접두사 옆에는 불가

### 5.2 실수 리터럴

```java
double pi = 3.141592653589793;
float f = 3.14F;
double avogadro = 6.022e23;
```

- 실수 리터럴 기본 타입은 `double`
- `float`로 쓰려면 `F` 접미사 필요

### 5.3 문자/문자열 리터럴

```java
char c1 = 'A';
char c2 = '\n';
char c3 = '\uAC00'; // '가'
String s = "Java";
```

Escape 문자 예:
- `\n` 개행
- `\t` 탭
- `\\` 백슬래시
- `\"` 큰따옴표

### 5.4 boolean과 null 리터럴

```java
boolean ok = true;
String s = null;
```

`null`은 참조가 없음을 나타내는 특수 리터럴이다.

## 6. `var` (지역 변수 타입 추론)

Java 10부터 지역 변수 선언에서 타입 추론을 지원한다.

```java
var name = "kim";    // String
var count = 10;      // int
var rate = 0.1;      // double
```

핵심:
- 런타임 동적 타입이 아니다.
- 컴파일 시점에 고정 타입이 결정된다.

## 6.1 `var` 사용 제약

다음은 허용되지 않는다.

```java
// var x;          // 초기값 없어서 타입 추론 불가
// var y = null;   // null만으로 타입 결정 불가
```

`var`는 지역 변수에만 사용 가능하며, 필드 타입 선언에는 사용할 수 없다.

### 6.2 `var`를 써도 좋은 경우

1. 타입이 코드에서 명확할 때  
2. 제네릭 타입이 길어 선언이 장황할 때  
3. IDE 지원으로 실제 타입 확인이 쉬울 때

### 6.3 `var`를 피해야 하는 경우

1. 초기값만 보고 타입 의도가 불명확할 때  
2. 숫자 리터럴 타입 혼동 위험이 있을 때  
3. 팀 코드 스타일에서 명시 타입을 선호할 때

---

## 7. `final` + `var` 조합

지역 불변 변수 + 간결한 선언이 동시에 가능하다.

```java
final var users = java.util.List.of("A", "B");
```

하지만 가독성이 떨어지면 명시 타입이 더 낫다.

---

## 8. 자주 하는 실수

1. `final`이면 객체 내부 상태도 못 바꾼다고 오해  
2. `var`를 남용해서 타입 의도를 숨김  
3. 리터럴 접미사 누락(`float`에 `F` 누락, `long`에 `L` 누락)  
4. 8진수 리터럴(`012`)을 10진수로 착각  
5. 상수를 하드코딩으로 흩뿌려 변경 지점을 늘림

---

## 9. 실무 가이드

1. 의미 있는 고정값은 `static final` 상수로 모아라  
2. 상수명은 도메인 용어 중심으로 작성하라 (`MAX_LOGIN_RETRY`)  
3. `var`는 "짧아서"가 아니라 "명확해서" 사용하라  
4. 리터럴은 타입과 단위를 함께 표현하라 (`TIMEOUT_MS = 3_000`)  
5. 금액/정밀 계산은 리터럴 단계부터 `BigDecimal` 전략을 고려하라

---

## 10. 정리

- `final`은 값의 안정성을, 리터럴은 값 표현의 정확성을, `var`는 선언의 간결성을 제공한다.
- 세 기능은 목적이 다르므로 상황에 맞게 선택해야 한다.
- 특히 `final`의 의미(재대입 금지)와 `var`의 의미(컴파일 타임 타입 추론)를 혼동하지 않는 것이 중요하다.
